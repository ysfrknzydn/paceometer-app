# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

Paceometer: a real-time in-car pace/speed display, built as an installable Progressive Web App. Part of a research project (with Professor Helveston, GWU) testing whether a pace display (not just a speedometer) reduces speeding. The literature review, research plan, and design justifications live in a **sibling repo**, `/Users/yusuf/paceometer_lit_review` (GitHub `ysfrknzydn/paceometer`) — check `lit_review.md` and `paceometer_review.qmd` there before adding or changing any behavior-design decision (pace formula, display layout, zone thresholds), rather than inventing a rationale. Not every phrase in that repo's `research_plan.qmd` is literature-backed — some (e.g. "optimal zone") are this project's own unresolved design extensions, not cited findings; don't assume plan language implies a citation exists.

## Commands

There is no build step, no bundler, no package.json, no linter, and no test suite — this is intentional (zero-budget, plain HTML/CSS/JS).

- **Run locally**: `python3 -m http.server 8000`, then open `http://localhost:8000`. Must be served over HTTP(S) — opening `index.html` directly as a `file://` URL breaks the ES module imports. Geolocation works on `localhost` without HTTPS, but testing from a phone requires the deployed HTTPS URL (a phone hitting your laptop's LAN IP is not a secure context, so geolocation is blocked there).
- **Deploy**: push to `main`. GitHub Pages serves directly from the repo root on that branch — no CI/build step in between. Live at `https://ysfrknzydn.github.io/paceometer-app/`.
- **Sanity-check JS**: no test runner exists, so after editing any `js/*.js` file, at least syntax-check it (these are ES modules, so `node --check` needs the file to have an `.mjs`-recognized module context — copy to a `.mjs` file first, e.g. into the scratch dir, then `node --check that.mjs`).
- **Verify UI changes with a real screenshot**: this is a visual/mobile-layout-heavy app, and CSS that looks right in source can render wrong (see the `@media (orientation: landscape)` history). Don't just read the CSS — run a local server (`python3 -m http.server`) and drive it headlessly (e.g. Playwright, installed ad hoc into the scratch dir since it isn't a project dependency) to screenshot both portrait and landscape before calling a layout change done.
- **Test without driving**: tap the "Dev tools ▸" disclosure link under the trip controls (collapsed by default as of 2026-07-16 so it doesn't clutter the live view — no query param needed), then pick a profile (Full range, Residential, Inner City, Highway, Rural — `SIMULATED_DRIVE_PROFILES` in `js/app.js`) from the dropdown next to "Start Simulated Drive" to feed synthetic speed data through the exact same `handlePosition()` pipeline real GPS uses — exercises speed/pace/gauges/trip-recording end to end without leaving the couch. Must be removed before shipping to real study participants — see Pre-launch checklist below.
- **Database schema changes**: use versioned migrations via the Supabase CLI (`brew install supabase/tap/supabase`), not the old single-file `supabase/schema.sql` (deprecated, kept only for history). Workflow: `supabase migration new <description>` creates a timestamped file in `supabase/migrations/`; write the SQL there (additive changes as idempotent `alter table ... add column if not exists`, same reasoning as before — the live table already exists and won't be recreated); then `supabase db push` applies any pending migrations to the linked remote project. The CLI must be linked once per machine (`supabase login --token ...`, then `supabase link --project-ref ojhhlxmbawckknnpgmfj`) — see git history around 2026-07-14 for how that was set up if it needs redoing. `supabase db push` needs the database password (Supabase dashboard → Connect → Direct → password, or reset it there) passed via `-p`; it does NOT need Docker. (`supabase db pull`/`db dump` do need Docker for the shadow-database/pg_dump-version-matching machinery — avoid those; if you ever need a raw schema dump instead, `brew install postgresql@17` and use `pg_dump` directly against the session pooler host, since the direct-connection hostname is IPv6-only and this network doesn't resolve it.) Migration history is tracked in a remote table the CLI manages — `supabase migration list -p <password>` shows what's applied. Never type a Supabase access token or DB password directly into a shell command visible in a Claude Code session transcript (`! command`); write it to a local file outside the repo first (outside the `!` proxy, in a real terminal window) and read it via command substitution instead, e.g. `-p "$(cat ~/.some_token_file)"`.

## Architecture

**No framework, no bundler.** `index.html` loads a single entry point, `js/auth.js`, via `<script type="module">`; everything else is pulled in through ES module imports (including the Supabase client from `esm.sh`, a CDN that serves npm packages as ES modules — used instead of a local `node_modules` install since there's no build step).

**Module graph**: `auth.js` (entry) → imports `supabaseClient.js` (client init) and `app.js` (GPS/speed/trip logic, exports `startApp`/`stopApp`).

**Screen-gating pattern**: `index.html` has three top-level `.screen` divs — `#auth-screen`, `#app`, and `#settings-screen` (added 2026-07-16) — toggled via the `.hidden` class. `auth.js`'s `supabase.auth.onAuthStateChange` listener is the single source of truth for the *signed-in vs. signed-out* split — it calls `startApp()` (begins `watchPosition`) and shows `#app` when a session exists, and calls `stopApp()` (clears the GPS watch, resets trip state) and shows `#auth-screen` when it doesn't; both `showApp()`/`showAuth()` also force `#settings-screen` hidden so it can never stay visible across a sign-out. Don't add a second place that shows/hides `#auth-screen`/`#app`; route all auth-driven visibility changes through that listener. The `#app` ↔ `#settings-screen` toggle is intentionally a *separate*, independent mechanism (`settings-nav`/`settings-back` listeners in `js/app.js`) — the driver stays signed in the whole time, so it doesn't belong in the auth listener, same reasoning as the trip-summary inline swap below.

**Speed → pace pipeline** (`js/app.js`): `handlePosition` prefers `coords.speed` from the Geolocation API; when that's `null` (no GPS chip — true of every desktop browser, and occasionally a phone's first fix), it falls back to a Haversine distance/time delta between consecutive fixes. Either way, the *only* output that leaves `handlePosition` is a plain `mph` number — raw lat/lng from `lastPosition` is used transiently for the fallback calc and never stored or transmitted. Pace is computed live from that same `mph` value using `t = d/v` (Peer & Gamliel 2013, Formula 1: minutes to cover a fixed 10-mile reference distance) — this is the one formula in the codebase that's directly literature-validated; if you change the reference distance or the formula shape, that traceability is lost, so check the sibling lit-review repo first (see "What this is" above).

**GPS fallback noise guards, and a known-open gap** (`js/app.js`, `MAX_FIX_ACCURACY_METERS`/`MAX_PLAUSIBLE_MPH` in `handlePosition`): desktop Wi-Fi/IP-based positioning is coarse and jumps between fixes, so the Haversine fallback above can turn ordinary position noise into a large apparent speed. Two guards catch the extreme case: a fix isn't remembered as "last known position" for the delta calc unless its reported `coords.accuracy` is within 100m, and any resulting speed above 200mph is discarded outright rather than shown/recorded (2026-07-15, added after "huge numbers on refresh" during dev-server testing). **Not fully fixed**: two fixes can still differ by tens of meters from ordinary noise even while both individually pass the 100m accuracy gate — e.g. a stationary laptop can still show a spurious few-mph or double-digit reading. The real fix (not yet implemented, see `TODO.md`) is to only trust a delta as real movement when the apparent displacement exceeds the *combined* accuracy radius of the two fixes, not just each fix's individual accuracy. Much smaller-magnitude on a real phone (GPS accuracy ~5-10m vs. desktop's hundreds), but not zero risk there either — don't assume this is a dev-only artifact.

**Core Function/Core Loop: the zone indicator** (`js/app.js`): the app's build follows a 5-stage MVP funnel from `research_plan.qmd` (`@sec-mvp-funnel`) — Core Function, Core Loop, Accessory Features, Surface Area Check, Retention Hook — each stage gating what gets built next. `marginalSecondsSaved(mph)` answers the Core Function question directly: at the current speed, would going `ZONE_SPEED_INCREMENT_MPH` (+10mph) faster still save `ZONE_THRESHOLD_SECONDS` (60s) or more over the `PACE_REFERENCE_MILES` reference distance? That's the exact hyperbola argument from `paceometer_review.qmd`'s own illustrative tool, checkable against its worked examples. The live indicator is a 3-state red/yellow/green traffic light (`nextZoneState`/`ZONE_STATE_LABELS`/`setZoneDisplay` in `js/app.js`; `.zone-indicator.green/.yellow/.red` in `css/style.css`), not just 2-state — `ZONE_THRESHOLD_SECONDS` (60s, ~73mph) is the red/not-red line and is what `pct_time_in_zone` and the end-of-trip summary are computed from; `ZONE_NEARING_THRESHOLD_SECONDS` (120s, ~50mph, exactly double the first) is a purely cosmetic second boundary — not literature-derived, just lands on a clean number on the same hyperbola — that only splits the "still helps" region into green (clearly) and yellow (nearing the limit) for the live display. Each state shows a big word (SPEED STILL HELPS / NEARING THE LIMIT / SPEED WON'T HELP), a big color-matched number, and a caption — never a sentence to parse. `ZONE_HYSTERESIS_SECONDS` (±5s dead-zone as of 2026-07-16, halved from the original ±10s because the padding felt like too much in practice; applied independently at both boundaries) prevents flickering when GPS noise puts the driver right at ~50mph or ~73mph, and a brief `zone-flash` animation marks the moment the state actually changes (Core Loop's "display confirms the new state"). The zone is defined against a fixed reference speed, not the posted speed limit or crash-risk data — confirmed with the professor's collaborator (2026-07-15) as a deliberate simplification: speed-limit lookups have no free tier, and crash risk is too location-dependent to be a clean per-driver baseline, so the app leads with the time-savings zone now and treats fuel efficiency as a future direction once vehicle make/model data is in scope. If a future change wants `pct_time_in_zone` to key off the yellow/green split instead of the red/not-red line, that's a decision to raise explicitly, not make silently.

**Trip recording is a separate layer on top of the live display**: `recordSample()` only accumulates into `trip` while `recording` is true, i.e. between Start Trip and End Trip taps. The live speed/pace/zone readout runs continuously once signed in regardless of whether a trip is being recorded. `endTrip()` sends the aggregates to Supabase — `avg_speed_mph`/`max_speed_mph`/`min_speed_mph`, `sample_count`, `distance_miles` (integrated from `mph × Δt` between samples, not from lat/lng, so it doesn't violate the privacy invariant below), `avg_pace_seconds` (a genuinely separate running mean of per-sample pace, not derived from `avg_speed_mph` — mean-of-pace ≠ pace-of-mean-speed, so comparing the two after a drive is a sanity check on the pace formula itself), and `pct_time_in_zone` (Accessory Feature — the same hysteresis-corrected `zoneState` the live indicator computes, time-integrated over the trip; time below `PACE_MIN_SPEED_MPH` is excluded from both sides of the ratio rather than counted as "out of zone"). Never a location trace.

**End-of-trip summary** (`js/app.js`, `showTripSummary`/`hideTripSummary`/`endTrip`/`zoneCeilingMph`; `index.html`, `#trip-summary`): the Accessory Feature stage's "kept only if it reinforces the core loop" gate is why this is deliberately just the one number, no historical trends, nothing about the car. It's an inline swap: ending a trip hides `#readout`/`#trip-controls` and shows `#trip-summary` in their place — **explicitly kept as an inline swap, not a separate screen, during the Surface Area Check discussion (2026-07-16)**, since the plan counts "screens" as UX concepts rather than literal routes and there was no functional reason to change it. The live in-trip readout (`setTripZoneProgressDisplay`, `#trip-zone-progress`) was originally a "Trip: ±X vs 55mph" baseline-speed comparison that predated and contradicted this summary's zone framing — resolved 2026-07-15 by replacing it with a running "% of trip in zone so far" readout, later the same day joined by a second line (`#trip-zone-progress-time`) showing the running version of the seconds-behind-pace number described below. **Wording resolved 2026-07-16**: both that second line and the end-of-trip caption (`tripSummaryCaptionEl`) had used "behind the fastest pace," flagged as confusing ("fastest pace" reads as literal top speed; "behind" doesn't say behind what). Out of 4 drafted candidates, the user picked the zone+reference-speed hybrid: both now read "Xs behind the ~73mph efficient pace[, so far]," with the `~73` computed live as `Math.round(zoneCeilingMph())`, never hardcoded.

**The end-of-trip headline number itself was revised again later the same day (2026-07-15):** it originally showed that same "% of trip in zone" as a flat percentage ("X% of this trip, more speed would have helped"). Dogfooding a simulated highway drive (65-80mph, i.e. cruising right at/above the ~73mph zone ceiling) showed the problem: that trip reported 100%, which reads as "you should have gone even faster than 80mph" — exactly backwards from the app's point, and it doesn't distinguish a trip that was 6 seconds off an efficient pace from one that drove 25mph the whole time. Replaced with a concrete seconds value: `secondsBehindPace = inZoneSeconds − (inZoneMiles / zoneCeilingMph()) × 3600`, clamped to ≥0 — actual time spent, only during the portion of the trip where the zone state was green/yellow (not red), above what those same miles would have taken at the exact zone-ceiling speed (`zoneCeilingMph()`, solved algebraically from `ZONE_THRESHOLD_SECONDS`, not hardcoded). A highway-speed trip now reports a small number; a trip driven well under an efficient pace reports a larger one. `pct_time_in_zone` is unchanged and still saved to Supabase (useful for research analysis) — only the on-screen headline number changed. Full derivation and the literature-grounding question (none of these thresholds are literature-derived; only `t=d/v` itself is) are written out in README.md's "How the pace/zone math works" section for a non-AI reader.

**Settings screen** (`js/app.js`, `settings-nav`/`settings-back` listeners; `index.html`, `#settings-screen`; `css/style.css`, `.settings-*`): built 2026-07-16 as part of Surface Area Check, reachable via a gear icon (44×44px tap target — an earlier ~12×21px version was reported as both hard to tap and rendering flush against the viewport edge because `env(safe-area-inset-top, 16px)` resolves to a *defined* `0` on non-notch screens, so the fallback never applied; fixed with `max(env(...), 16px)`) on the live dashboard. Contains an "About Paceometer" section (plain-language pace formula, traffic-light meaning, end-of-trip number, an explicit "doesn't know the speed limit" caveat) and a "Privacy" section (no raw GPS ever leaves the phone, what summary fields are saved, RLS scope) — both held to the same literature-backed-vs-design-choice honesty standard as this file and README.md, just in plainer language for a non-technical driver. Sign-out moved here from its old floating position directly on `#app`.

**Portrait/landscape layout**: the app is meant to be mounted either way on a dash, so `css/style.css` gives landscape its own explicit font sizes/spacing in `@media (orientation: landscape)` rather than reusing the portrait `vw`-based formula scaled down — they looked comparable on paper but wildly different on an actual phone, which is why real screenshots (see Commands) matter here more than for most CSS changes. `index.html` also carries a small inline script working around a known iOS Safari bug where the layout viewport fails to recompute after rotation (nudges `initial-scale` off 1 and back on `orientationchange`).

**One more palette: GWU's official colors (2026-07-21, fourth same-day follow-up)**: added as the 18th theme option (`data-theme="gwu"`) at the user's request — the one theme here with a real institutional identity (Professor Helveston's GWU) rather than a picked-for-fun palette. **Supabase access model** (see `supabase/migrations/`): the project was created with "Automatically expose new tables" turned **off**, so any *new* table needs an explicit `grant select, insert on public.<table> to authenticated;` or the API will reject all access to it regardless of RLS policies — this is a common silent-failure trap when adding tables. RLS policies scope both `insert` and `select` to `auth.uid() = user_id`, so each signed-in user can only write/read their own rows; nothing is granted to the `anon` role at all, so unauthenticated requests are rejected before RLS even runs. Only the developer (via the Supabase dashboard or a service-role key, which must never be added to client code) can see the full table across users. The anon key hardcoded in `js/supabaseClient.js` is safe to have committed — RLS is the actual access boundary, not key secrecy.

**Privacy invariant to preserve in any future change**: no raw GPS coordinates ever get sent to Supabase or logged anywhere off-device — only derived scalar metrics. This is a standing project requirement (from the professor meeting that set this architecture), not just current scope; if a change would add a lat/lng field to any outgoing payload, that's a decision to raise explicitly, not make silently.

## Pre-launch checklist

Before shipping this app to real study participants (as opposed to solo dev use), remove:

- **Simulated drive dev tool** (`js/app.js`, search for `DEV TOOL`) — lets speed/pace/trip-recording be exercised indoors via fake driving profiles instead of real GPS.

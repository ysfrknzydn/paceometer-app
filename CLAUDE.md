# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

Paceometer: a real-time in-car pace/speed display, built as an installable Progressive Web App. Part of a research project (with Professor Helveston, GWU) testing whether a pace display (not just a speedometer) reduces speeding. The literature review, research plan, and design justifications live in a **sibling repo**, `/Users/yusuf/paceometer_lit_review` (GitHub `ysfrknzydn/paceometer`) — check `lit_review.md` and `paceometer_review.qmd` there before adding or changing any behavior-design decision (pace formula, display layout, zone thresholds), rather than inventing a rationale. Not every phrase in that repo's `research_plan.qmd` is literature-backed — some (e.g. "optimal zone") are this project's own unresolved design extensions, not cited findings; don't assume plan language implies a citation exists.

## Commands

There is no build step, no bundler, no package.json, no linter, and no test suite — this is intentional (zero-budget, plain HTML/CSS/JS).

- **Run locally**: `python3 -m http.server 8000`, then open `http://localhost:8000`. Must be served over HTTP(S) — opening `index.html` directly as a `file://` URL breaks the ES module imports. Geolocation works on `localhost` without HTTPS, but testing from a phone requires the deployed HTTPS URL (a phone hitting your laptop's LAN IP is not a secure context, so geolocation is blocked there).
- **Deploy**: push to `main`. GitHub Pages serves directly from the repo root on that branch — no CI/build step in between. Live at `https://ysfrknzydn.github.io/paceometer-app/`.
- **Sanity-check JS**: no test runner exists, so after editing any `js/*.js` file, at least syntax-check it (these are ES modules, so `node --check` needs the file to have an `.mjs`-recognized module context — copy to a `.mjs` file first, e.g. into the scratch dir, then `node --check that.mjs`).
- **Verify UI changes with a real screenshot**: this is a visual/mobile-layout-heavy app, and CSS that looks right in source can render wrong (see the `@media (orientation: landscape)` history). Don't just read the CSS — run a local server (`python3 -m http.server`) and drive it headlessly (e.g. Playwright, installed ad hoc into the scratch dir since it isn't a project dependency) to screenshot both portrait and landscape before calling a layout change done.
- **Test without driving**: click "Start Simulated Drive" in the running app (a dev tool, visible directly in the normal UI, no query param needed) to feed a synthetic 0→120mph profile through the exact same `handlePosition()` pipeline real GPS uses — exercises speed/pace/gauges/trip-recording end to end without leaving the couch. Must be removed before shipping to real study participants — see Pre-launch checklist below.
- **Database schema changes**: use versioned migrations via the Supabase CLI (`brew install supabase/tap/supabase`), not the old single-file `supabase/schema.sql` (deprecated, kept only for history). Workflow: `supabase migration new <description>` creates a timestamped file in `supabase/migrations/`; write the SQL there (additive changes as idempotent `alter table ... add column if not exists`, same reasoning as before — the live table already exists and won't be recreated); then `supabase db push` applies any pending migrations to the linked remote project. The CLI must be linked once per machine (`supabase login --token ...`, then `supabase link --project-ref ojhhlxmbawckknnpgmfj`) — see git history around 2026-07-14 for how that was set up if it needs redoing. `supabase db push` needs the database password (Supabase dashboard → Connect → Direct → password, or reset it there) passed via `-p`; it does NOT need Docker. (`supabase db pull`/`db dump` do need Docker for the shadow-database/pg_dump-version-matching machinery — avoid those; if you ever need a raw schema dump instead, `brew install postgresql@17` and use `pg_dump` directly against the session pooler host, since the direct-connection hostname is IPv6-only and this network doesn't resolve it.) Migration history is tracked in a remote table the CLI manages — `supabase migration list -p <password>` shows what's applied. Never type a Supabase access token or DB password directly into a shell command visible in a Claude Code session transcript (`! command`); write it to a local file outside the repo first (outside the `!` proxy, in a real terminal window) and read it via command substitution instead, e.g. `-p "$(cat ~/.some_token_file)"`.

## Architecture

**No framework, no bundler.** `index.html` loads a single entry point, `js/auth.js`, via `<script type="module">`; everything else is pulled in through ES module imports (including the Supabase client from `esm.sh`, a CDN that serves npm packages as ES modules — used instead of a local `node_modules` install since there's no build step).

**Module graph**: `auth.js` (entry) → imports `supabaseClient.js` (client init) and `app.js` (GPS/speed/trip logic, exports `startApp`/`stopApp`).

**Screen-gating pattern**: `index.html` has two top-level `.screen` divs, `#auth-screen` and `#app`, toggled via the `.hidden` class. `auth.js`'s `supabase.auth.onAuthStateChange` listener is the single source of truth for which screen shows — it calls `startApp()` (begins `watchPosition`) when a session exists and `stopApp()` (clears the GPS watch, resets trip state) when it doesn't. Don't add a second place that shows/hides these screens; route all visibility changes through that listener.

**Speed → pace pipeline** (`js/app.js`): `handlePosition` prefers `coords.speed` from the Geolocation API; when that's `null` (some devices/first fix), it falls back to a Haversine distance/time delta between consecutive fixes. Either way, the *only* output that leaves `handlePosition` is a plain `mph` number — raw lat/lng from `lastPosition` is used transiently for the fallback calc and never stored or transmitted. Pace is computed live from that same `mph` value using `t = d/v` (Peer & Gamliel 2013, Formula 1: minutes to cover a fixed 10-mile reference distance) — this is the one formula in the codebase that's directly literature-validated; if you change the reference distance or the formula shape, that traceability is lost, so check `lit_review_repo` first.

**Core Function/Core Loop: the zone indicator** (`js/app.js`): the app's build follows a 5-stage MVP funnel from `research_plan.qmd` (`@sec-mvp-funnel`) — Core Function, Core Loop, Accessory Features, Surface Area Check, Retention Hook — each stage gating what gets built next. `marginalSecondsSaved(mph)` answers the Core Function question directly: at the current speed, would going `ZONE_SPEED_INCREMENT_MPH` (+10mph) faster still save `ZONE_THRESHOLD_SECONDS` (60s) or more over the `PACE_REFERENCE_MILES` reference distance? That's the exact hyperbola argument from `paceometer_review.qmd`'s own illustrative tool, checkable against its worked examples. The live indicator (`setZoneDisplay`) shows the state as a big word (SPEED STILL HELPS / SPEED WON'T HELP), a big color-matched number, and a caption — never a sentence to parse. `ZONE_HYSTERESIS_SECONDS` (±10s dead-zone) prevents the state from flickering when GPS noise puts the driver right at the ~73mph boundary, and a brief `zone-flash` animation marks the moment the state actually flips (Core Loop's "display confirms the new state"). The zone is defined against a fixed reference speed, not the posted speed limit or crash-risk data — confirmed with the professor's collaborator (2026-07-15) as a deliberate simplification: speed-limit lookups have no free tier, and crash risk is too location-dependent to be a clean per-driver baseline, so the app leads with the time-savings zone now and treats fuel efficiency as a future direction once vehicle make/model data is in scope.

**Trip recording is a separate layer on top of the live display**: `recordSample()` only accumulates into `trip` while `recording` is true, i.e. between Start Trip and End Trip taps. The live speed/pace/zone readout runs continuously once signed in regardless of whether a trip is being recorded. `endTrip()` sends the aggregates to Supabase — `avg_speed_mph`/`max_speed_mph`/`min_speed_mph`, `sample_count`, `distance_miles` (integrated from `mph × Δt` between samples, not from lat/lng, so it doesn't violate the privacy invariant below), `avg_pace_seconds` (a genuinely separate running mean of per-sample pace, not derived from `avg_speed_mph` — mean-of-pace ≠ pace-of-mean-speed, so comparing the two after a drive is a sanity check on the pace formula itself), and `pct_time_in_zone` (Accessory Feature — the same hysteresis-corrected `zoneInZone` state the live indicator computes, time-integrated over the trip; time below `PACE_MIN_SPEED_MPH` is excluded from both sides of the ratio rather than counted as "out of zone"). Never a location trace.

**End-of-trip summary** (`js/app.js`, `showTripSummary`/`hideTripSummary`; `index.html`, `#trip-summary`): the Accessory Feature stage's "kept only if it reinforces the core loop" gate is why this is deliberately just the one number — % of the trip spent in the zone — with no historical trends, no vs-speed-baseline comparison, nothing about the car. It's an inline swap: ending a trip hides `#readout`/`#trip-controls` and shows `#trip-summary` in their place, no separate screen (that's a Surface Area Check decision, not made yet). The legacy live "Trip: ±X vs 55mph" readout (`setTripTimeSavedDisplay`, `TRIP_BASELINE_SPEED_MPH`) predates this and still uses a baseline-speed comparison the summary deliberately avoids — known inconsistency, flagged in `app.js`, not yet resolved.

**Portrait/landscape layout**: the app is meant to be mounted either way on a dash, so `css/style.css` gives landscape its own explicit font sizes/spacing in `@media (orientation: landscape)` rather than reusing the portrait `vw`-based formula scaled down — they looked comparable on paper but wildly different on an actual phone, which is why real screenshots (see Commands) matter here more than for most CSS changes. `index.html` also carries a small inline script working around a known iOS Safari bug where the layout viewport fails to recompute after rotation (nudges `initial-scale` off 1 and back on `orientationchange`).

**Supabase access model** (see `supabase/migrations/`): the project was created with "Automatically expose new tables" turned **off**, so any *new* table needs an explicit `grant select, insert on public.<table> to authenticated;` or the API will reject all access to it regardless of RLS policies — this is a common silent-failure trap when adding tables. RLS policies scope both `insert` and `select` to `auth.uid() = user_id`, so each signed-in user can only write/read their own rows; nothing is granted to the `anon` role at all, so unauthenticated requests are rejected before RLS even runs. Only the developer (via the Supabase dashboard or a service-role key, which must never be added to client code) can see the full table across users. The anon key hardcoded in `js/supabaseClient.js` is safe to have committed — RLS is the actual access boundary, not key secrecy.

**Privacy invariant to preserve in any future change**: no raw GPS coordinates ever get sent to Supabase or logged anywhere off-device — only derived scalar metrics. This is a standing project requirement (from the professor meeting that set this architecture), not just current scope; if a change would add a lat/lng field to any outgoing payload, that's a decision to raise explicitly, not make silently.

## Pre-launch checklist

Before shipping this app to real study participants (as opposed to solo dev use), remove:

- **Simulated drive dev tool** (`js/app.js`, search for `DEV TOOL`) — lets speed/pace/trip-recording be exercised indoors via a fake profile instead of real GPS. Remove that whole block plus the `#simulate-btn` button and `#simulate-progress` bar in `index.html` and their `.simulate-btn`/`.simulate-progress*` rules in `css/style.css`.
- **`TRIP_BASELINE_SPEED_MPH` in `js/app.js`** — the live "time saved this trip" readout compares actual trip time to a placeholder 55mph baseline that isn't tied to a speed limit or a literature source. Decide on (or justify) a real baseline before this number is something participants should trust, or drop the feature.

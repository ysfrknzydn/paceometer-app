- **GPS fallback speed bug — partially fixed (same day, follow-up)** — reported as "huge numbers on refresh, takes a few tries" during dev-server testing. Root cause: desktop browsers have no GPS chip, so `coords.speed` is essentially always null and the app falls back to computing speed from the distance/time delta between fixes; desktop Wi-Fi/IP-based positioning is coarse (accuracy routinely hundreds+ meters) and jumps between fixes, so a large apparent jump over a short time delta produced a physically impossible speed with nothing catching it. Added two guards in `handlePosition` (`js/app.js`): a fix isn't remembered as "last known position" for the delta calc unless its reported accuracy is within `MAX_FIX_ACCURACY_METERS` (100m), and any resulting speed above `MAX_PLAUSIBLE_MPH` (200) is discarded. This stopped the triple-digit/absurd readings, verified with Playwright. **Not fully fixed** — see the open item below, found by the user immediately after this fix landed.

## Next session: remaining pipeline stages

- **GPS fallback still shows spurious low speed while genuinely stationary (open, found 2026-07-15, right after the fix above landed)** — the `MAX_FIX_ACCURACY_METERS`/`MAX_PLAUSIBLE_MPH` guards stop the extreme/implausible readings (triple-digit+ mph), but the user reports still seeing "a few mph" or "double-digit" speeds on the dev server while the laptop is sitting still. Root cause is a different flavor of the same underlying problem: two consecutive fixes can differ by tens of meters just from ordinary position noise even when both fixes individually pass the 100m accuracy gate — desktop Wi-Fi positioning jitters within its own uncertainty radius even at rest, and the fallback formula (`distance / elapsedSeconds`) treats *any* apparent displacement as real motion, with nothing currently distinguishing "actually moved" from "GPS noise within the margin of error." Standard fix direction for later (not implemented — user asked to log this, not fix it now): only trust a delta as real movement if the apparent displacement exceeds the *combined* accuracy radius of the two fixes (`haversineMeters(a, b) > a.accuracy + b.accuracy`), treating smaller apparent movement as effectively stationary (mph = 0) rather than computing a speed from it. Likely much less visible on a real phone — GPS accuracy is roughly 5-10m vs. desktop Wi-Fi positioning's hundreds of meters, so the same at-rest jitter translates to a much smaller spurious speed, and a lot of it would likely already fall under `PACE_MIN_SPEED_MPH` (5) and simply not show a pace/zone reading — but per the user, "worse on desktop, not zero risk on the phone," so still worth the combined-accuracy fix eventually rather than assuming it's a dev-only artifact.
